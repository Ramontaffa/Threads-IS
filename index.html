02 — processo & memória compartilhada
Processo: AUDIO-THREADS
PID único · espaço de memória próprio · 3 threads internas
AudioState — memória compartilhada entre todas as threads
0x…a0 globalPlay atomic<bool>
0x…a8 currentFrame atomic<size_t>
0x…b0 programRunning atomic<bool>
0x…b8 totalFrames size_t (read-only)
0x…c0 sampleRate int (read-only)
0x…d0 tracks[ ] vector<Track>
↳ 0x…e0 tracks[i].isPlaying atomic<bool>
↳ 0x…f0 tracks[i].pcmData vector<float> (read-only)

Campos atomic são modificados em runtime por múltiplas threads. Campos read-only são escritos uma única vez pela main thread antes das outras threads existirem — por isso não precisam de proteção.

03 — as três threads do processo
Thread 01
Main Thread
Criada automaticamente pelo SO quando o processo inicia. Executa o main() do programa.
// criação automática
int main() { // ← já é uma thread }
  • Carrega .mp3 e decodifica pra PCM (AudioSetup)
  • Inicializa AudioState com todos os campos
  • Chama rtAudioSetup e startAudioStream
  • Cria a UI thread e fica em join() esperando ela terminar
  • Após join, finaliza stream e encerra
Thread 02
Audio Callback Thread
Criada pelo sistema operacional quando RtAudio abre a stream. Não é um std::thread — o SO a gerencia.
// RtAudio instrui o SO:
startAudioStream(&state);
// SO cria thread dedicada
  • Acorda a cada ~5ms quando o DAC precisa de dados
  • Lê globalPlay.load(relaxed)
  • Lê tracks[i].isPlaying.load(relaxed)
  • Mistura samples das faixas ativas no buffer
  • Executa currentFrame.fetch_add(1, relaxed)
  • Nunca bloqueia — qualquer bloqueio = glitch de áudio
executando a cada ~5.8ms @ 44100Hz / buffer 256
Thread 03
UI Thread
No modo terminal: criada com std::thread. No modo GUI: loop roda na própria main thread via runGui().
// modo terminal:
std::thread uiThread(uiLoop, &state);
uiThread.join();
  • Lê input do usuário (teclado ou ImGui)
  • PLAY/PAUSE: globalPlay.store(!val, relaxed)
  • Toggle faixa: isPlaying.store(!val, relaxed)
  • RESET: currentFrame.store(0, relaxed)
  • Lê currentFrame.load pra exibir progresso
  • Ao sair: programRunning.store(false)

As três threads existem simultaneamente durante o uso do mixer. Não há coordenação direta entre elas — a comunicação toda passa pelos campos atomic do AudioState.

05 — mixingCallback — coração da audio thread
mixingCallback — AudioEngine.cpp
⏱ executado a cada ~5ms pelo SO 
int mixingCallback(void* outputBuffer, void* inputBuffer,
    unsigned int nFrames, ..., void* userData) {
    float* out = static_cast<float*>(outputBuffer);
    AudioState* state = static_cast<AudioState*>(userData);

    for (unsigned int i = 0; i < nFrames; i++) {
        // ↓ leitura atômica — sem bloqueio
        if (!state->globalPlay.load(memory_order_relaxed)) {relaxed
            *out++ = 0.0f; // silêncio
            continue;
        }

        size_t frameIndex = state->currentFrame.load(memory_order_relaxed);relaxed

        for (unsigned int c = 0; c < state->channels; c++) {
            float mixedSample = 0.0f;
            int activeTracks = 0;

            for (auto& track : state->tracks) {
                // ↓ cada faixa tem seu próprio atomic
                if (track.isPlaying.load(memory_order_relaxed)) {relaxed
                    mixedSample += track.pcmData[frameIndex * channels + c];
                    activeTracks++;
                }
            }

            if (activeTracks > 1)
                mixedSample /= (float)activeTracks;
            *out++ = mixedSample;
        }

        // ↓ operação atômica de incremento — indivisível no hardware
        state->currentFrame.fetch_add(1, memory_order_relaxed);fetch_addrelaxed
    }

    return 0; // sinaliza RtAudio pra continuar
}
06 — por que não há mutex, semáforo ou lock
ⓧ COM MUTEX — não usado aqui
// hipotético
audioMutex.lock(); ← pode bloquear!
// ... mixar áudio ...
audioMutex.unlock();
✗ Thread de áudio fica bloqueada esperando a UI soltar o mutex
✗ Bloqueio de 10ms = 440 samples perdidos @ 44100 Hz
✗ Resultado: click, pop, ou silêncio no áudio
✗ Jitter imprevisível — depende do scheduler do SO
✓  COM ATOMIC — lock-free — usado aqui
// sem lock, sem espera
globalPlay.load(relaxed); // instantâneo
currentFrame.fetch_add(1, relaxed);
✓ Operação executada diretamente pelo processador (instrução LOCK x86)
✓ Nenhuma thread fica parada esperando outra
✓ memory_order_relaxed: atômico + mais rápido possível
✓ Seguro aqui porque UI e callback são independentes entre si
07 — AudioState — quem lê e quem escreve cada campo
campos atômicos — modificados em runtime
globalPlay atomic<bool> UI escreve
globalPlay atomic<bool> audio lê
currentFrame atomic<size_t> audio fetch_add
currentFrame atomic<size_t> UI lê (display)
tracks[i].isPlaying atomic<bool> UI escreve
tracks[i].isPlaying atomic<bool> audio lê
programRunning atomic<bool> UI store(false)
campos read-only — escritos uma vez pela main thread
totalFrames size_t sem proteção necessária
sampleRate int sem proteção necessária
channels unsigned int sem proteção necessária
tracks[i].pcmData vector<float> sem proteção necessária
tracks[i].trackName string sem proteção necessária
Múltiplas threads podem ler simultaneamente sem lock — só há race condition quando há escrita concorrente
08 — detalhe técnico — move constructor em Track
include/AudioState.h — Track não é copiável porque std::atomic não é copiável 
// ✓ Move constructor — ensina o vector como mover Track na memória
Track(Track&& other) noexcept
    : pcmData(std::move(other.pcmData)),
      isPlaying(other.isPlaying.load(memory_order_relaxed)),
      trackName(std::move(other.trackName)),
      originalSampleRate(other.originalSampleRate) { }

// ✗ Cópia deletada — std::atomic não permite cópia
Track(const Track&) = delete;
Track& operator=(const Track&) = delete;
